## 核心能力

### 1. 场景→命令映射（28 个预定义场景）

| 分类 | 场景数 | 示例 |
|------|--------|------|
| 📱 应用性能 | 8 | 冷启动、页面跳转、触控响应、动画、IPC、内存、综合、FFRT |
| 🖥️ 系统性能 | 6 | CPU调度、IRQ中断、磁盘IO、电源、内核全面、内存总线 |
| 🌐 网络与通信 | 3 | 网络请求、蓝牙、分布式协同 |
| 🎬 多媒体 | 3 | 音频、视频/相机、图形渲染管线 |
| 🔧 系统服务 | 5 | 通知、传感器、多模态输入、安全/密钥、文件/存储 |
| 📸 快照模式 | 2 | 文本+自定义tag / 二进制+bgsrv默认tag |
| 📼 录制模式 | 1 | 长时间二进制录制 |

每个场景包含：tag 组合、缓冲区大小、时长、输出格式、命令模板、描述。

### 2. Agent 对话内菜单（核心工作流）

**问题**：开发者口头描述需求 → Agent 需要映射到精确的 hitrace 命令。

**方案**：Agent 不启动交互式 shell，而是在对话中呈现两级菜单：

```
开发者: "抓个trace"
  → Agent 展示分类菜单（1-7）
    → 开发者选 "1"
      → Agent 展示子菜单（8个应用场景）
        → 开发者选 "1"（冷启动）
          → Agent 后台启动 trace + 同轮发送操作提示
            → 等待完成 → 拉取文件
```

**一步直达**：开发者说"冷启动trace" → 直接匹配 JSON 执行，跳过菜单。

### 3. 设备操作提示（同轮同步）

**关键发现**：trace 启动和操作提示必须在同一轮对话发出，否则开发者看到提示时窗口已过半。

```
❌ 旧: 先 exec trace → 等返回 → 下一轮发提示（延迟 3-5s）
✅ 新: background exec trace + 同轮回复发 ⚡ 提示（0 延迟）
```

12 个交互场景各有对应操作提示（如冷启动→"请在设备上打开目标应用"）。

### 4. 平台兼容

- macOS: 自动检测 DevEco Studio SDK 中的 hdc 路径（4 个候选位）
- 设备: 支持 `hitrace -t` / `--trace_begin` / `--start_bgsrv` / `--record` 全模式
- 路径: 文本 `/data/local/tmp/`，二进制 `/data/log/hitrace/`，自动适配

## 关键技术决策

| 决策 | 理由 |
|------|------|
| 使用 `--json` 而非在 SKILL.md 内嵌数据 | 场景数据由 CLI 工具统一管理，SKILL.md 只写工作流 |
| Agent 对话菜单 vs 启动交互式 shell | Agent 无终端给开发者交互，必须在对话内呈现 |
| `background: true` 启动 trace | 确保提示和 trace 同时到达开发者 |
| 场景数据用 shell 数组 + JSON 输出 | 一组数据两套消费（Agent JSON + 开发者 CLI TUI） |
| SKILL.md 顶层放「标准工作流」 | Agent 进入 skill 第一眼就知道该怎么走 |

## 已知限制 & 后续迭代方向

1. **场景不可自定义** — 当前 28 个场景是硬编码的，开发者不能临时加 tag/改时长
   - 方向: 在菜单中增加"自定义"选项，Agent 询问 tag 和时长后执行
2. **无 trace 质量反馈** — 抓完后不判断数据是否有价值（如 46KB 的空 trace）
   - 方向: 文件大小 < 阈值时提示"可能未捕获到有效数据"
3. **设备操作提示依赖人工** — 无法检测开发者是否真的操作了设备
   - 方向: 考虑用倒计时 + 确认机制替代
4. **不支持多设备** — 当前只连一台设备
   - 方向: 多设备时让开发者选 target
5. **二进制格式支持有限** — 主要走文本格式，快照 bgsrv 拉取不稳定
   - 方向: 完善 .sys 拉取逻辑 + SmartPerf 启动引导
6. **场景覆盖不全** — 缺部分内核 tag（erofs、hmfs 等）和互联场景
   - 方向: 补全 tag 覆盖面
